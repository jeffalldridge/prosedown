prosedown

Write books in Markdown. Compile to EPUB.

Like Fountain is for screenplays, ProseDown is for ebooks: plain text that compiles to a complex format. Two required fields, no custom syntax, no lock-in.

The whole tool, in seven lines

$ cat > on-simplicity.md <<'EOF'
---
title: "On Simplicity"
author: "Jane Smith"
---

# On Simplicity

The simplest things are often the most profound.
EOF

$ prosedown build on-simplicity.md on-simplicity.epub

That's a valid EPUB 3.3. Passes EPUBCheck. Opens in Apple Books, Kobo, Calibre, anything that reads ebooks. Two required frontmatter fields. One Markdown file. No configuration.

What happens when you run build

   your project                  prosedown build                  EPUB 3.3
   ┌────────────────┐         ┌──────────────────┐          ┌──────────────┐
   │ book.md        │         │   parse YAML     │          │ META-INF/    │
   │ ┌────────────┐ │         │   resolve files  │          │   container.xml │
   │ │frontmatter │ │  ──▶    │   render *.md    │   ──▶    │ OEBPS/       │
   │ ├────────────┤ │         │   build OPF/NCX  │          │   chapters/  │
   │ │   body     │ │         │   pack ZIP       │          │   nav.xhtml  │
   │ └────────────┘ │         └──────────────────┘          │   content.opf│
   │ 01-foo.md      │            internal             │ mimetype     │
   │ 02-bar.md      │                                  │              │
   │ cover.jpg      │           deterministic           └──────────────┘
   └────────────────┘             EPUBCheck-clean

Everything is auto-detected: cover by filename, CSS by filename, chapter order by filename. Same source produces the same bytes — diff-clean under git, reproducible builds work.

What deconstruct does (the other direction)

   existing.epub              prosedown deconstruct            readable Markdown
   ┌────────────────┐         ┌──────────────────┐          ┌──────────────┐
   │  .epub (ZIP)   │         │   read OPF       │          │ book.md      │
   │   ┌──────────┐ │         │   classify roles │          │   (frontmatter)│
   │   │ XHTML    │ │  ──▶    │   walk spine     │   ──▶    │ 000-cover.md │
   │   │ + assets │ │         │   XHTML→Markdown │          │ 001-chap1.md │
   │   │ + OPF    │ │         │   extract images │          │ 002-chap2.md │
   │   └──────────┘ │         └──────────────────┘          │ images/      │
   └────────────────┘            best-effort              │ css/         │
                              documented normalization     └──────────────┘

Building is the primary use case — stable and deterministic. Deconstruction is best-effort with documented normalization. The goal is readable Markdown an author would want to edit, not a lossless archive of the original EPUB.

Why this exists

Most ebook tools fall into one of three buckets:

Vellum, Scrivener		Power tools

ProseDown sits between: plain Markdown source files an author can open in any editor — Obsidian, VS Code, iA Writer, TextEdit — that compile to a professional EPUB by convention, with no configuration.

Five design principles

  1. One file is enough. A single .md with two YAML lines is a valid project. The barrier is zero.
  2. Convention over configuration. Cover auto-detected. CSS auto-detected. Chapter order from filenames. Authors shouldn't configure what can be inferred.
  3. Standard everything. CommonMark Markdown. Standard YAML. No custom syntax. Lock-in is the enemy of adoption.
  4. Two directions. Build is primary, deterministic. Deconstruct is best-effort, practical.
  5. Non-destructive. Opens cleanly in Obsidian, Hugo, Jekyll. A format that fights your other tools isn't a format worth adopting.
