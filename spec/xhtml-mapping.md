# ProseDown XHTML Mapping

What HTML each Markdown element produces. This is the contract between the spec and any compiler implementation.

All output is XHTML5 (HTML serialized as XML — self-closing void elements, lowercase tags, quoted attributes).

---

## Block Elements

### Headings

| Markdown | XHTML |
|----------|-------|
| `# Title` | `<h1>Title</h1>` |
| `## Section` | `<h2>Section</h2>` |
| `### Subsection` | `<h3>Subsection</h3>` |
| Through `######` | Through `<h6>` |

Setext headings (`Title\n=====`) produce the same output as ATX headings.

Heading IDs are auto-generated by slugifying the heading text: lowercase, spaces to hyphens, strip punctuation. Example: `## The Journey Begins` → `<h2 id="the-journey-begins">The Journey Begins</h2>`

### Paragraphs

| Markdown | XHTML |
|----------|-------|
| `Text here.` | `<p>Text here.</p>` |
| (blank line between) | Separate `<p>` elements |

No classes. The default CSS handles indentation via `p + p { text-indent: 1.5em; }`.

### Blockquotes

```markdown
> This is a quote.
>
> — Attribution
```

```html
<blockquote>
  <p>This is a quote.</p>
  <p>— Attribution</p>
</blockquote>
```

### Lists

```markdown
- Item one
- Item two
  - Nested item
```

```html
<ul>
  <li>Item one</li>
  <li>Item two
    <ul>
      <li>Nested item</li>
    </ul>
  </li>
</ul>
```

Ordered lists use `<ol>` and `<li>`. Nested lists nest as expected.

### Definition Lists

```markdown
Term
: Definition text
```

```html
<dl>
  <dt>Term</dt>
  <dd>Definition text</dd>
</dl>
```

Multiple definitions per term and multiple terms per definition block are supported per Pandoc/PHP Markdown Extra behavior.

### Code Blocks

````markdown
```python
def hello():
    print("world")
```
````

```html
<pre><code class="language-python">def hello():
    print("world")</code></pre>
```

The language hint becomes a `class="language-X"` attribute. If no language is specified, just `<pre><code>`.

### Horizontal Rules (Scene Breaks)

```markdown
---
```

```html
<hr />
```

The default CSS renders this as a typographic ornament (`* * *`), not a visible line. The HTML stays semantic — the CSS handles presentation.

`***` and `___` also produce `<hr />`.

### Tables

```markdown
| Column A | Column B |
|----------|----------|
| Value 1  | Value 2  |
```

```html
<table>
  <thead>
    <tr>
      <th>Column A</th>
      <th>Column B</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Value 1</td>
      <td>Value 2</td>
    </tr>
  </tbody>
</table>
```

## Inline Elements

| Markdown | XHTML |
|----------|-------|
| `**bold**` | `<strong>bold</strong>` |
| `*italic*` | `<em>italic</em>` |
| `***bold italic***` | `<strong><em>bold italic</em></strong>` |
| `` `code` `` | `<code>code</code>` |
| `[text](url)` | `<a href="url">text</a>` |
| `~~strikethrough~~` | `<del>strikethrough</del>` (if supported) |

### Images

**Without title (plain image):**

```markdown
![A mountain trail](images/trail.jpg)
```

```html
<img src="images/trail.jpg" alt="A mountain trail" />
```

**With title (captioned figure):**

```markdown
![A mountain trail](images/trail.jpg "The trail above Redfish Lake")
```

```html
<figure>
  <img src="images/trail.jpg" alt="A mountain trail" />
  <figcaption>The trail above Redfish Lake</figcaption>
</figure>
```

The `title` attribute is NOT included on the `<img>` element — it goes to `<figcaption>` only. This prevents screen reader confusion from duplicate descriptions.

### Links (Cross-File)

```markdown
[Chapter 2](02-chapter-2.md)
[The Summit](03-chapter-3.md#the-summit)
```

```html
<a href="chapter-2.xhtml">Chapter 2</a>
<a href="chapter-3.xhtml#the-summit">The Summit</a>
```

The `.md` extension is replaced with `.xhtml` during compilation. Fragment identifiers are preserved.

## Footnotes

```markdown
This needs a source.[^1]

[^1]: See Smith, 2024, p. 42.
```

**Inline reference:**

```html
<sup><a epub:type="noteref" role="doc-noteref" href="#fn-1" id="fnref-1">1</a></sup>
```

**Footnote definition (end of chapter):**

```html
<aside epub:type="footnote" role="doc-footnote" id="fn-1">
  <p><a href="#fnref-1" role="doc-backlink">1.</a> See Smith, 2024, p. 42.</p>
</aside>
```

Footnotes use both `epub:type` (for legacy reading systems) and DPUB-ARIA `role` (for modern accessibility). The backlink allows readers to return to their place.

## Smart Typography (Build-Time)

Applied to text nodes after Markdown parsing. Does NOT apply inside `<code>`, `<pre>`, URLs, or HTML attributes.

| Source | Output |
|--------|--------|
| `"text"` | `"text"` (U+201C / U+201D) |
| `'text'` | `'text'` (U+2018 / U+2019) |
| `don't` | `don't` (U+2019 — right single quote) |
| `'tis` | `'tis` (U+2019 — right single quote, not left) |
| `--` | `–` (U+2013 — en-dash) |
| `---` | `—` (U+2014 — em-dash) |
| `...` | `…` (U+2026 — ellipsis) |

Language-aware: `language: fr` produces `« text »`, `language: de` produces `„text"`.

## Document Structure

Each chapter file compiles to one XHTML content document. The structure:

```html
<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en">
<head>
  <title>Chapter Title</title>
  <link rel="stylesheet" type="text/css" href="../css/style.css" />
</head>
<body epub:type="bodymatter">
  <section epub:type="chapter" role="doc-chapter">
    <h1>Chapter Title</h1>
    <p>First paragraph...</p>
    <p>Second paragraph...</p>
  </section>
</body>
</html>
```

### Role-Based Wrappers

| `role:` value | `<body>` epub:type | `<section>` epub:type | `<section>` ARIA role |
|---------------|--------------------|-----------------------|-----------------------|
| `chapter` | `bodymatter` | `chapter` | `doc-chapter` |
| `frontmatter` | `frontmatter` | *(varies by slug)* | *(none — no generic ARIA for frontmatter)* |
| `backmatter` | `backmatter` | *(varies by slug)* | *(none — no generic ARIA for backmatter)* |
| `part` | `bodymatter` | `part` | `doc-part` |

For known slugs, the compiler MAY use more specific `epub:type` values:

| Slug | `epub:type` |
|------|-------------|
| `copyright` | `imprint` |
| `dedication` | `dedication` |
| `foreword` | `foreword` |
| `preface` | `preface` |
| `prologue` | `prologue` |
| `epilogue` | `epilogue` |
| `appendix` | `appendix` |
| `bibliography` | `bibliography` |
| `glossary` | `glossary` |
| `colophon` | `colophon` |
| `afterword` | `afterword` |
| `about-the-author` | `contributors` |

## Raw HTML Passthrough

Raw HTML in Markdown source is preserved and normalized:

- Lowercase all tag names
- Quote all attribute values
- Self-close void elements (`<br />`, `<hr />`, `<img ... />`)
- Strip `<script>` elements entirely
- Strip event-handler attributes (`onclick`, `onload`, etc.)
- Preserve `class`, `id`, `epub:type`, `role`, `data-*` attributes
- Preserve inline `style` attributes (but discourage them)

Example input:

```markdown
<div class="scripture">
<p><sup>16</sup> For God so loved the world...</p>
</div>
```

Passes through as-is (with XML normalization if needed).

## Navigation Document (nav.xhtml)

Generated from chapter titles and headings up to `toc-depth`:

```html
<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en">
<head>
  <title>Table of Contents</title>
</head>
<body>
  <nav epub:type="toc" role="doc-toc">
    <h1>Table of Contents</h1>
    <ol>
      <li><a href="chapter-1.xhtml">The Trailhead</a></li>
      <li><a href="chapter-2.xhtml">Above the Trees</a>
        <ol>
          <li><a href="chapter-2.xhtml#the-ridge">The Ridge</a></li>
        </ol>
      </li>
      <li><a href="chapter-3.xhtml">The Summit</a></li>
    </ol>
  </nav>
</body>
</html>
```

Parts create nesting:

```html
<ol>
  <li><a href="part-1.xhtml">Part One: The Ascent</a>
    <ol>
      <li><a href="chapter-1.xhtml">The Beginning</a></li>
      <li><a href="chapter-2.xhtml">The Middle</a></li>
    </ol>
  </li>
  <li><a href="part-2.xhtml">Part Two: The Descent</a>
    <ol>
      <li><a href="chapter-3.xhtml">The Turning</a></li>
    </ol>
  </li>
</ol>
```

Frontmatter files are excluded from the TOC unless `toc: true` is set. Backmatter files are included by default.
